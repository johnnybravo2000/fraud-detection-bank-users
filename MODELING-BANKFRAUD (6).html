Dictionary

  • income (numeric): Annual income of the applicant (in decile form). Ranges between [0.1, 0.9].
  • name_email_similarity (numeric): Metric of similarity between email and applicants name. Higher values represent higher similarity. Ranges between [0, 1].
  • prev_address_months_count (numeric): Number of months in previous registered address of the applicant, i.e.the applicants previous residence, if applicable. Ranges between [1, 380] months (-1 is a missing value).
  • current_address_months_count (numeric): Months in currently registered address of the applicant. Ranges between [1, 429] months (-1 is a missing value).
  • customer_age (numeric): Applicants age in years, rounded to the decade. Ranges between [10, 90] years.
  • days_since_request (numeric): Number of days passed since application was done. Ranges between [0, 79] days.
  • intended_balcon_amount (numeric): Initial transferred amount for application. Ranges between [16, 114] (negatives are missing values).
  • payment_type (categorical): Credit payment plan type. 5 possible (annonymized) values.
  • zip_count_4w (numeric): Number of applications within same zip code in last 4 weeks. Ranges between [1, 6830].
  • velocity_6h (numeric): Velocity of total applications made in last 6 hours i.e., average number of applications per hour in the last 6 hours. Ranges between [175, 16818].
  • velocity_24h(numeric): Velocity of total applications made in last 24 hours i.e.,average number of applications per hour in the last 24 hours. Ranges between [1297, 9586]
  • velocity_4w (numeric): Velocity of total applications made in last 4 weeks, i.e., average number of applications per hour in the last 4 weeks. Ranges between [2825, 7020].
  • bank_branch_count_8w (numeric): Number of total applications in the selected bank branch in last 8 weeks. Ranges between [0, 2404].
  • date_of_birth_distinct_emails_4w (numeric): Number of emails for applicants with same date of birth in last 4 weeks. Ranges between [0, 39].
  • employment_status (categorical): Employment status of the applicant. 7 possible (annonymized) values.
  • credit_risk_score (numeric): Internal score of application risk. Ranges between [191, 389].
  • email_is_free (binary): Domain of application email (either free or paid).
  • housing_status (categorical): Current residential status for applicant. 7 possible (annonymized) values.
  • phone_home_valid (binary): Validity of provided home phone.
  • phone_mobile_valid (binary): Validity of provided mobile phone.
  • bank_months_count (numeric): How old is previous account (if held) in months. Ranges between [1, 32] months (-1 is a missing value).
  • has_other_cards (binary): If applicant has other cards from the same banking com- pany.
  • proposed_credit_limit (numeric): Applicants proposed credit limit. Ranges between [200, 2000].
  • foreign_request (binary): If origin country of request is different from banks country. - source (categorical): Online source of application. Either browser (INTERNET) or app (TELEAPP).
  • session_length_in_minutes (numeric): Length of user session in banking website in minutes. Ranges between [1, 107] minutes (-1 is a missing value).
  • device_os (categorical): Operative system of device that made request. Possible values are: Windows, macOS, Linux, X11, or other.
  • keep_alive_session (binary): User option on session logout.
  • device_distinct_emails (numeric): Number of distinct emails in banking website from the used device in last 8 weeks. Ranges between [1, 2] emails (-1 is a missing value). - device_fraud_count (numeric): Number of fraudulent applications with used device. Ranges between [0, 1].
  • month (numeric): Month where the application was made. Ranges between [0, 7].
  • fraud_bool (binary): If the application is fraudulent or not.

Steps for the classification tasks

  1. Feature selection: The aim is to select the best variables to build our models with.
  2. Data partitioning
  3. Class balancing
  4. Deploying the model

1. Feature selection

The following code was implemented to run the Boruta package and perform the feature selection. These lines have been excluded from the R-chunk due to its high processing time.

Model Selection and Results

Model Selection:

We decided to use Decision Trees as the main model in our project to identify fraudulent bank transactions. Accurately detecting fraudulent transactions while reducing false positives and false negatives was the main emphasis of the study question. Decision trees are frequently employed in fraud detection because of their interpretability and capacity to handle both numerical and categorical data, according to our literature review. This type of model is characterized by its transparency, allowing us to visualize decision paths and comprehend why a transaction is regarded as fraudulent. Logistic Regression and Random Forest models were also addressed, to evaluate the performance and accuracy cross models to determine the optimal model to implement.

Approach And Selection:

The steps in our approach were as follows:

Data Preprocessing: Categorical variables such as transaction type and location were previously encoded and missing data identified with \(-1\) in the dataset was excluded, reducing the initial records to \(124,260\) representing a \(87.57\) reduction respectively. Class imbalance was managed by creating new data points for the minority class (fraud_bool = 1 - a record classified as fraudulent) by using Synthetic Minority Oversampling Technique (SMOTE).

Feature engineering: We focused on features that are pertinent to fraud detection, such as phone home valid, employment status, payment type and credit risk score.

Model Selection and Training: Using performance criteria such as accuracy, precision, recall, and F1 score, we compared the models of Random Forest, Logistic Regression, and Decision Trees. The decision tree was chosen due to its ease of interpretation, accuracy of 78.21%, and ability to strike a fair balance between precision and generalization without going overboard.

The Decision Tree model is ideal because it matches the structure of our data and produces dependable findings while facilitating simple interpretation. Its simple decision-making process makes it a perfect model for situations where knowledge and transparency are crucial, particularly in a commercial setting.

Evaluating the Models Performance: Advantages and Weaknesses:

Assessment of Performance: With 78.21% accuracy, 77.36% sensitivity, and 79.06% specificity, the Decision Tree model proved to be an excellent classifier of both fraudulent and non-fraudulent transactions.

Advantages:

Interpretability: The interpretability of Decision Trees is one of their main benefits. Clear decision paths that are simple to understand for stakeholders that are not technical are provided by the model.

Simplicity: The concept is beneficial in real-time contexts where speed and transparency are crucial because it is computationally efficient and easy to apply.

Versatility: Without requiring significant data preprocessing, decision trees can handle both numerical and categorical variables.

Weaknesses:

Overfitting: Although Decision Trees are strong tools, improper tuning can lead to overfitting. In order to prevent over-complexity, we defined a limit depth and used cross-validation.

Lower Accuracy: The Decision Tree model offers a trade-off between performance and interpretability, but it does not reach the best accuracy when compared to more sophisticated models like Random Forest.

Empirical Results:

With an accuracy of 78.21%, the Decision Tree model did a good job at identifying fraudulent transactions. It provided findings that were easy to understand for each decision the model made, striking a compromise between performance and transparency.

Principal Results: With a sensitivity of 77.36%, the model successfully identified a sizable percentage of non-fraudulent transactions. With a specificity of 79.06%, the model was able to detect most fraudulent transactions with accuracy, lowering the possibility of fraud going undiscovered. The models interpretability makes it useful in comprehending and explaining why particular transactions are tagged as fraudulent, even though its performance metrics are not as high as Random Forests.

Conclusion: In summary, the Decision Tree model is a solid choice for real-world business applications where transparency is essential because it strikes a reasonable compromise between accuracy and interpretability. When a model achieves 100% accuracy, it is likely overfitting, which means it has become insensitive to noise in the training set and may not perform well on fresh data. An excessively complicated model or data leakage may be the cause of this lack of generality. Due to the inherent uncertainty in data, it is unreasonable to anticipate 100% accuracy in business; instead, it is more vital to focus on robust performance indicators like precision and recall.

Even while it falls short of Random Forests flawless accuracy, it offers a more approachable and realistic method of detecting fraud, which is crucial for preserving public confidence in the financial sector.
